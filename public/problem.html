001. Fix the Skewed Join

Medium Meta Netflix Uber

Problem Statement

You have two tables with a heavily skewed join key. The top 1% of customers account for 50% of all orders, causing extreme executor skew during joins.

The following query times out due to data skew:

SELECT c.customer_name, c.segment, COUNT(*) as order_count, SUM(o.amount) as total_spent FROM orders o JOIN customers c ON o.customer_id = c.customer_id GROUP BY c.customer_name, c.segment;

Your task: Rewrite the query to handle the skew efficiently. The result must match the expected output exactly.

Tables

-- orders: 100M rows, skewed on customer_id -- Top 10 customers have 50M orders orders ( order_id STRING, customer_id BIGINT, amount DOUBLE, order_date DATE ) -- customers: 1M rows, evenly distributed customers ( customer_id BIGINT, customer_name STRING, segment STRING )

Constraints

  • Solution must complete in under 30 seconds
  • Cannot modify underlying table structure
  • Result must match expected output exactly

Hints

The issue is that a few customer_ids have millions of rows. When Spark shuffles data for the join, these heavy keys all land on the same executor, causing it to run out of memory or take forever.

The customers table is only 1M rows. Could you avoid the shuffle entirely by broadcasting it?

For extreme skew, you can "salt" the join key - add a random suffix to distribute heavy keys across multiple partitions. You'll need to "explode" the smaller table to match.

Approach 1: Broadcast Join

If the small table fits in memory, broadcast it to avoid shuffle entirely.

SELECT /*+ BROADCAST(c) */ c.customer_name, c.segment, COUNT(*) as order_count, SUM(o.amount) as total_spent FROM orders o JOIN customers c ON o.customer_id = c.customer_id GROUP BY c.customer_name, c.segment;

When to use: Small table < 10GB (configurable via spark.sql.autoBroadcastJoinThreshold)

Approach 2: Salting

For extreme skew where broadcast won't work, distribute heavy keys across partitions.

WITH salted_orders AS ( SELECT *, CONCAT(customer_id, '_', FLOOR(RAND() * 10)) as salted_key FROM orders ), exploded_customers AS ( SELECT c.*, CONCAT(c.customer_id, '_', salt.n) as salted_key FROM customers c CROSS JOIN (SELECT explode(sequence(0, 9)) as n) salt ) SELECT c.customer_name, c.segment, COUNT(*) as order_count, SUM(o.amount) as total_spent FROM salted_orders o JOIN exploded_customers c ON o.salted_key = c.salted_key GROUP BY c.customer_name, c.segment;

Approach 3: Adaptive Query Execution

Let Spark handle it automatically (Spark 3.0+).

SET spark.sql.adaptive.enabled = true; SET spark.sql.adaptive.skewJoin.enabled = true; -- Original query now works SELECT ...
